Quiero que construyas una plataforma web llamada "Fierro Escuela".

## Contexto del producto
Es una plataforma de capacitación para clientes e internos de Fierro (fierro.com.ar),
un ERP para librerías y editoriales. El objetivo es que los usuarios puedan aprender
a implementar Fierro de forma autogestionada a través de cursos organizados en módulos
de video con material de apoyo.

## Stack
- Next.js 16 (App Router, TypeScript, Tailwind CSS 4)
- Supabase (auth, base de datos, storage)
- TipTap (editor rich text)
- React Hook Form + Zod 4
- @dnd-kit (drag & drop para reordenar módulos)
- lucide-react (íconos)

## Branding
Basado en el sistema de diseño oficial de Fierro:
- Fondo: #FFFFFF
- Navbar / headers: #1B3A7A (azul oscuro)
- Acento principal (botones CTA, highlights): #F5A623 (naranja)
- Acento secundario (links, íconos activos): #1E6FBF (azul medio)
- Texto principal: #212121
- Texto secundario: #666666
- Bordes y separadores: #E0E0E0
- Tipografía: Roboto (importar desde Google Fonts)
- Estética: limpia, profesional, clara — similar a un LMS corporativo

## Roles de usuario
- `admin`: equipo de Fierro, gestiona contenido y usuarios desde panel admin
- `student`: clientes de Fierro, acceden solo a los cursos que el admin les habilitó

## Productos disponibles (5 cursos)
Cada curso corresponde a un producto de Fierro:
1. Librería
2. Editorial
3. FierroGO Librería
4. FierroGO Editorial
5. Distribuidora

## Estructura de datos

Cursos (courses):
- id, title, product_key ('libreria' | 'editorial' | 'fierrogo_libreria' |
  'fierrogo_editorial' | 'distribuidora'), description, order_index,
  published (boolean), created_at, updated_at

Módulos (modules):
- id, course_id (FK → courses), title, description (rich text/html),
  video_url (YouTube embed), order_index (integer), published (boolean),
  created_at, updated_at

Archivos adjuntos (module_files):
- id, module_id (FK → modules), name, file_url, created_at

Perfiles (profiles, extiende auth.users de Supabase):
- id (FK → auth.users), email, full_name, role ('admin' | 'student'), created_at

Acceso a cursos por usuario (user_courses):
- id, user_id (FK → profiles), course_id (FK → courses),
  assigned_at, assigned_by (FK → profiles)
- UNIQUE(user_id, course_id)

Progreso por usuario (user_progress):
- id, user_id (FK → profiles), module_id (FK → modules),
  completed (boolean), completed_at
- UNIQUE(user_id, module_id)

Storage bucket: `attachments` (público)

## Páginas — Alumno

/login
- Formulario email + contraseña
- Logo Fierro centrado, fondo blanco, navbar azul

/escuela
- Dashboard del alumno
- Muestra solo los cursos que el admin le asignó
- Si no tiene ninguno: mensaje "Tu acceso está siendo configurado por el equipo de Fierro"
- Cada curso: título, descripción, progreso general (X/Y módulos completados)

/escuela/[course_id]
- Listado de módulos del curso (solo published = true)
- Cada módulo: título, check si completado
- Progreso visual del curso

/escuela/[course_id]/[module_id]
- Título del módulo
- Video embed de YouTube (responsive, 16:9)
- Descripción en rich text
- Lista de archivos adjuntos descargables
- Botón "Marcar como completado" → upsert en user_progress
- Navegación anterior / siguiente módulo

## Páginas — Admin

/admin
- Listado de cursos con cantidad de módulos y alumnos asignados
- Botón "Nuevo curso"
- Acceso rápido a gestión de usuarios

/admin/cursos/nuevo
- Formulario: título, product_key (selector), descripción, published

/admin/cursos/[id]
- Editar datos del curso
- Listado de módulos con drag & drop para reordenar
- Botón "Nuevo módulo"
- Publicar/despublicar curso

/admin/cursos/[id]/modulos/nuevo
- Formulario: título, descripción (editor TipTap), video_url, published
- Subida de archivos adjuntos (múltiples PDFs)

/admin/cursos/[id]/modulos/[module_id]
- Igual que nuevo pero precargado
- Sección para ver y eliminar archivos ya subidos

/admin/usuarios
- Listado de todos los usuarios registrados
- Columnas: nombre, email, fecha de registro, cursos asignados
- Buscador por nombre o email

/admin/usuarios/nuevo
- El admin crea la cuenta del estudiante (no hay registro público)
- Campos: nombre completo, email, contraseña temporal

/admin/usuarios/[id]
- Ver perfil del usuario
- Asignar / quitar cursos (checkboxes por cada curso disponible)
- Ver progreso del usuario por curso

## Protección de rutas
- /escuela/* → solo usuarios autenticados
- /admin/* → solo usuarios con role = 'admin'
- Si no autenticado → redirigir a /login
- Student intenta /admin → redirigir a /escuela
- Login con admin → redirigir a /admin
- Login con student → redirigir a /escuela

## Pasos para construir en orden
1. Crear el proyecto con create-next-app (TypeScript + Tailwind + App Router)
2. Instalar todas las dependencias listadas arriba
3. Configurar cliente Supabase para browser y server (SSR)
4. Generar el SQL completo del schema para que yo lo ejecute en Supabase
5. Configurar Roboto desde Google Fonts en el layout global
6. Crear el sistema de auth: /login con el branding de Fierro
7. Crear middleware/proxy de protección de rutas
8. Construir /escuela (dashboard del alumno)
9. Construir /escuela/[course_id] y /escuela/[course_id]/[module_id]
10. Construir /admin/usuarios (listado + asignación de cursos)
11. Construir /admin/cursos y gestión de módulos

Arrancá por el paso 1. Avisame cuando necesites las keys de Supabase (paso 3).
Antes de escribir cualquier componente, preguntame si tengo alguna preferencia de
estructura o confirmame que podés avanzar con las decisiones que tomaste.

NOTAS TÉCNICAS IMPORTANTES
================================================================================

- Next.js 16 renombró middleware.ts a proxy.ts con export nombrado `proxy`
- Zod 4 usa z.infer<T> (alias de z.output<T>)
- TipTap requiere `immediatelyRender: false` para evitar errores de hidratación
- El cliente Supabase del proxy debe usar service_role para omitir RLS
  en verificaciones de rol
- No hay registro público: el admin crea las cuentas de los estudiantes
